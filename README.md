# Gauge Reader System

Веб-система приёма RTMP-видеопотоков с камер (в т.ч. телефона), обработки кадров и регистрации показаний **аналоговых** и **цифровых** измерительных приборов средствами компьютерного зрения.

В составе решения:

| Компонент | Назначение |
|-----------|------------|
| **nginx-rtmp** | Приём RTMP-потоков (`live/<ключ>`), статистика по HTTP |
| **Backend** (FastAPI, PostgreSQL, Redis) | API, пользователи, логгеры, распознавание, хранение данных |
| **Frontend** (React, Vite) | Вход, дашборд, настройка логгеров и калибровка |

---

## Что нужно на компьютере

- **Docker** с поддержкой Compose:
  - **Windows / macOS**: [Docker Desktop](https://www.docker.com/products/docker-desktop/)
  - **Linux**: пакеты `docker-ce` и `docker-compose-plugin` (или Docker Desktop для Linux)

Другие языки и СУБД ставить не обязательно — всё поднимается в контейнерах.

## Клонирование и переход в каталог проекта

Репозиторий на GitHub называется **`diplom`** (пример):

```bash
git clone https://github.com/SverdlovSemen/diplom.git
cd diplom
```

Дальше все команды выполняйте из каталога, в котором лежит файл **`docker-compose.yml`** (если compose лежит внутри подпапки — перейдите в неё, например `cd gauge-reader-system`).

## Запуск основного приложения (без тестовых видеопотоков)

Сборка образов и запуск всех сервисов в foreground (логи в терминале):

```bash
docker compose up --build
```

Фоновый режим:

```bash
docker compose up --build -d
```

Остановка:

```bash
docker compose down
```

### Куда заходить в браузере после запуска

| Сервис | Адрес |
|--------|--------|
| Веб-интерфейс | http://localhost:5173 |
| API (Swagger) | http://localhost:8000/docs |
| Статистика RTMP (nginx) | http://localhost:8080/stat |

Порты при необходимости можно изменить переменными окружения в `.env` или в вызове compose (см. `docker-compose.yml`: `BACKEND_PORT`, `FRONTEND_PORT`, `RTMP_PORT`, и т.д.).

## Учётные записи (вход в веб-интерфейс)

Значения по умолчанию задаются в **`docker-compose.yml`** в секции `backend` → `environment` (можно переопределить через `.env` или переменные окружения хоста):

| Роль | Переменные | Значения по умолчанию |

## Запуск с тестовыми видеопотоками (без телефона)

Сначала должен быть запущен основной стек (`docker compose up`). Затем можно включить **профили** Compose — дополнительные контейнеры с FFmpeg.

### Цифровой тестовый поток → логгер `logger-2`

Генерируется синтетическое видео: **чёрный фон и крупные белые цифры** (меняющееся число). Отдельной папки с такими же фотографиями в репозитории нет — имитация идёт средствами FFmpeg внутри контейнера.

```bash
docker compose --profile test-stream up -d
```

Поток публикуется на **`logger-2`** (см. `docker-compose.yml`, сервис `ffmpeg-test`).

Чтобы остановить только тестовый publisher:

```bash
docker compose --profile test-stream stop ffmpeg-test
```

(или `docker compose down` остановит всё.)

### Аналоговый тестовый поток (стрелка) → логгер `logger-1`

Нужны кадры в `test_images/analog_sequence/` (если их нет — сгенерируйте один раз на машине с Python):

```bash
cd scripts
python generate_analog_sequence.py
cd ..
```

Запуск контейнера с циклической подстановкой кадров:

```bash
docker compose --profile test-stream-analog up -d
```

Подсказки по калибровке для этого набора — в разделе [Настройка analog](#настройка-analog-кратко) и в файле `test_images/analog_sequence/calibration_logger1.json`.

### Запуск «всё сразу» с профилем

Пример одной командой с цифровым тестом:

```bash
docker compose --profile test-stream up --build -d
```

Аналогично для аналога добавьте профиль `test-stream-analog` (оба профиля можно указать через запятую в одной команде, если нужны оба потока).

## Трансляция с телефона (Prism Live Studio)

Телефон и компьютер с Docker должны быть в **одной локальной сети** (один Wi‑Fi роутер). На ПК должны быть открыты порты RTMP (**1935**) и при необходимости веб-интерфейса — см. файрвол ОС.

### Узнать IP-адрес компьютера в этой сети

- **Windows** (PowerShell или cmd):

  ```text
  ipconfig
  ```

  Найдите активный адаптер Wi‑Fi или Ethernet и строку **IPv4-адрес** (например `192.168.1.57` или `192.168.100.18`).

- **Linux**:

  ```bash
  hostname -I
  ```

  или `ip -br a`

- **macOS**:

  ```bash
  ipconfig getifaddr en0
  ```

Адрес может меняться при смене сети или после переподключения Wi‑Fi — перед каждым сеансом проверяйте текущее значение.

### Настройка Prism Live Studio

Интерфейс приложения может слегка отличаться по версии; общая схема такая:

1. Откройте **Prism Live Studio** на телефоне.
2. Выберите режим **RTMP / Прямая трансляция / Custom RTMP** (название раздела зависит от версии).
3. Задайте параметры публикации:
   - **URL сервера (RTMP URL)** — без ключа приложения, только до приложения `live`, например:

     `rtmp://192.168.1.57:1935/live`

     Подставьте **ваш** IPv4 вместо `192.168.1.57`.

   - **Ключ трансляции / Stream key** — должен совпадать с **ключом потока логгера** в системе, например **`logger-1`** или **`logger-2`** (эти логгеры можно создать или использовать уже заданные в интерфейсе).

   Некоторые приложения объединяют это в одну строку:

   `rtmp://192.168.1.57:1935/live/logger-1`

   Тогда поле «ключ» может быть пустым или дублировать последний сегмент — ориентируйтесь на подсказки Prism: главное, чтобы итоговый RTMP-путь совпадал с **`rtmp://<IP>:1935/live/<ключ>`**.

4. **Название трансляции** в Prism может быть любым — на сервере важны URL и ключ.

5. Убедитесь, что в веб-интерфейсе создан логгер с тем же **stream key**, запущена обработка/настройки по необходимости.

Проверка, что поток доходит до сервера: страница **http://localhost:8080/stat** на машине с Docker — должен отображаться активный поток.

## Тест без второго устройства: камера на распечатанное изображение

Для **аналогового** прибора можно распечатать или открыть на экране компьютера кадры со стрелкой и навести на них камеру телефона. Исходные кадры после генерации лежат в папке:

**`test_images/analog_sequence/`** — файлы `frame_01.jpg` … `frame_08.jpg` (генерация: `python scripts/generate_analog_sequence.py`).

Для имитации **цифрового** табло с белыми цифрами на чёрном фоне в проекте используется встроенный тестовый FFmpeg-поток (профиль **`test-stream`**) или трансляция того же типа с телефона; отдельного набора статичных фото «как analog_sequence» для цифрового режима в репозитории может не быть — ориентир для проверки цифры в облаке: контейнер **`ffmpeg-test`** (чёрный фон + `drawtext`). Дополнительно в **`test_images/README.txt`** описан файл **`gauge.jpg`** как вспомогательный тестовый кадр для локальных скриптов.

## Кратко об интерфейсе и параметрах

- **Dashboard** — общий обзор и быстрый доступ к разделам.
- **Loggers** — список измерительных точек (логгеров), привязка к **ключу RTMP-потока** (`logger-1`, `logger-2`, …).
- **Logger setup** — для каждого логгера: тип прибора (analog/digital), область интереса (**ROI**), калибровка шкалы (для analog — точки центра/мин/макс и масштаб), интервал опроса (**sample_interval_sec**), проверка качества (**Quality gate**, **Test recognize**).

Смысл основных сообщений качества: успешное распознавание vs предупреждения (например, стрелка не найдена); подробный регламент для analog — ниже.

---

## Настройка analog (кратко)

1. Нажмите **Refresh snapshot** и дождитесь чёткого кадра.
2. Шаг **ROI**: выделите только шкалу и стрелку.
3. **Center** — ось стрелки; **Min point** / **Max point** — направления на минимум и максимум шкалы.
4. **Scale values** (`min_value`, `max_value`) — пределы шкалы прибора.
5. **Save config** → **Test recognize** → при необходимости **Quality x6**.

Критерии: в **Quality gate** статус **PASS**, без частых **`needle_not_found`**, без ложного «залипания» на 0%/100%.

Если качество не проходит — сузьте ROI, переставьте точки, уберите блики и дрожание камеры.

Для тестового набора кадров analog используйте **`test_images/analog_sequence/calibration_logger1.json`** и ROI на весь кадр `{"x":0,"y":0,"w":640,"h":480}`; интервал опроса порядка **8** секунд удобен под смену кадров раз в 8 сек.

## Структура репозитория

- `backend/` — API, модели, миграции  
- `frontend/` — UI  
- `nginx/` — RTMP и HTTP для статуса  
- `docker/` — образы вспомогательных сервисов (тестовые потоки)

## Примечание для Windows (Docker Desktop)

На Docker Desktop с WSL2 часть сетевых сценариев отличается от «чистого» Linux; для тестовых контейнеров FFmpeg в проекте используется обычная bridge-сеть Docker и имя хоста **`nginx-rtmp`** (см. комментарии в `docker-compose.yml`).
